//overview

A fully modular RNG and case-opening framework for Roblox. Built around a weighted tier rolling engine with real pity tracking, a complete inventory and currency system, timed global events, and secure DataStore saving. Every system is server-authoritative and exploit-proof. All client logic is intentionally dumb, it fires requests and displays results. All rolls, validation, and data mutations happen on the server.

//showcase
//features
Weighted RNG Engine
Tier rolling uses relative weights so odds never need to sum to 100. A luck multiplier scales rarer tiers independently. Includes simulation and odds utilities for testing weight tables before shipping.
Pity System
Per-tier pity counters are tracked per case per player. When a player hits the cap for a tier, the next roll guarantees it. Counters persist across sessions via DataStore.
Inventory and Selling
Items stack per player. Players can sell individual items or all duplicates at once for coins. Sell value is determined by tier. Equipped items are tracked per slot.
Currency System
Supports coins and gems independently. All balance changes go through CurrencyService with hard cap validation. Clients never touch currency values directly.
Global Events
Timed server events can boost luck, discount case costs, or double sell payouts. Events auto-expire and notify all clients on start and end. New joining players receive the current event state immediately.
DataStore
Player data loads on join, saves on leave, and autosaves every 60 seconds. A BindToClose handler saves all connected players on server shutdown. New data fields are reconciled automatically for existing players.
//case opening flow
Every case open follows this exact 9-step pipeline on the server. The client receives the result only after all steps complete.
1. Validate request
→
2. Charge currency
→
3. Check pity
→
4. Roll tier
→
5. Roll item
→
6. Award item
→
7. Update pity
→
8. Update stats
→
9. Return result
If an item pool or roll fails after currency is charged, the player is automatically refunded. The client then plays the reveal animation and fires ClaimReward, which the server validates before confirming.

//configuration
Cases and items
Open CaseDefinitions in ReplicatedStorage/Shared to define your cases and the items inside each tier. Each case needs a unique id, a name, a cost, a currency type, an image asset id, and a table of items per tier. Item ids must be unique across all cases, use a case-specific prefix on every id to avoid collisions.
Tiers and weights
Open TierConfig in ReplicatedStorage/Shared to adjust tier weights, colors, pity caps, and sell values. The default tiers are Common, Uncommon, Rare, Epic, Legendary, Mythic, and Brainrot. Weights are relative and do not need to sum to 100.
Sounds
Open CaseController in StarterPlayerScripts/Controllers and find the SOUNDS table near the top of the file. Replace each rbxassetid://0 with your own sound asset ids.
open
Plays when the player clicks the open button, before the result is revealed
common
Plays on reveal for Common and Uncommon tier results
rare
Plays on reveal for Rare and Epic tier results
legendary
Plays on reveal for Legendary and Mythic tier results
brainrot
Plays on reveal for Brainrot tier results
▸
To add a sound for a new tier, add a new key to SOUNDS and a matching entry to TIER_SOUND_MAP just below it in the same file.
UI element names
The controllers reference your UI elements by name using dot-separated paths from PlayerGui. Match the names in your ScreenGuis to the paths used in each controller, or update the paths in the controller to match your own naming.
CaseController
References CaseGui.RevealFrame, CaseGui.OpenButton, CaseGui.PityFrame, and HudGui.EventBanner
InventoryController
References InventoryGui.ScrollFrame, InventoryGui.SellDupesButton, and InventoryGui.CloseButton. Requires an ItemCardTemplate Frame inside the ScrollFrame
CurrencyController
References HudGui.CoinLabel and HudGui.GemLabel
ClientMain
References HudGui.InventoryButton, HudGui.StarterCaseButton, HudGui.PremiumCaseButton, and CaseGui.CloseButton
Events
Open EventService in ServerScriptService/Services to customise the event pool. Adjust durations, multipliers, and which cases each event targets. Set AUTO_EVENT_INTERVAL to control how often a random event fires automatically, or set it to nil to disable automatic events entirely and trigger them manually.
Inventory
Open InventoryService in ServerScriptService/Services to adjust inventory behaviour. The MAX_STACK constant near the top of the file controls how many of the same item a player can hold at once. The default is 9999. Sell values per tier are defined in TierConfig under the sellValue field rather than in InventoryService directly.
//remotes reference
OpenCase
RemoteFunction
Client invokes to open a case. Returns the full result table including item, tier, balances, and pity summary.
GetCaseData
RemoteFunction
Client invokes when opening the case UI. Returns odds, pity progress, balances, and case definition.
GetInventory
RemoteFunction
Client invokes when opening the inventory UI. Returns the full inventory and equipped tables.
GetBalances
RemoteFunction
Client invokes on join and on demand. Returns current coin and gem balances.
ClaimReward
RemoteEvent
Client fires after the reveal animation finishes. Server validates a pending reward exists before confirming.
RewardGranted
RemoteEvent
Server fires to the client after ClaimReward is validated. Contains final item, tier, balances, and pity data.
SellItem
RemoteEvent
Client fires with an item id to sell one instance. Server validates ownership before removing and paying out.
SellDupes
RemoteEvent
Client fires to sell all duplicate items at once, keeping one of each. Server handles the full sweep.
EquipItem
RemoteEvent
Client fires with an item id and slot string. Server validates ownership before equipping.
UnequipSlot
RemoteEvent
Client fires with a slot string to clear that slot.
EventNotify
RemoteEvent
Server fires to all clients when a global event starts or ends. Payload includes event name, description, and duration.
//services
DataService
Foundation of the entire system. All other services read and write player data through DataService only. Never access DataStore directly from any other service.
DataService.Get(player)
Returns the live data table for a player. All mutations on this reference are reflected immediately.
DataService.GetKey(player, key)
Returns a specific field from a player's data table.
DataService.Set(player, key, value)
Sets a specific field in a player's data. Does not save immediately - autosave and leave save handle persistence.
DataService.Save(player)
Force-saves a player's data immediately. Use sparingly as DataStore has rate limits.
RNGService
Pure stateless roll engine. No player data, no dependencies. Weights are relative and do not need to sum to any specific value.
RNGService.RollTier(tierTable, luckMultiplier)
Rolls a tier from a weighted table. Luck multiplier boosts all non-common tiers proportionally.
RNGService.RollItem(itemPool)
Rolls a single item from a pool of items. Items within a tier can have individual weights.
RNGService.GetOdds(tierTable)
Returns a table of tier names to percentage chances. Use this to display odds in the case UI.
RNGService.Simulate(tierTable, rolls, luck)
Simulates N rolls and returns tier frequency. Use in Studio to verify weight tables before shipping.
CurrencyService
Handles all coin and gem mutations. Never set currency directly via DataService. All operations validate the currency type and amount. Balances are capped server-side to prevent overflow.
CurrencyService.GetBalance(player, type)
Returns a player's current balance for "currency" or "gems".
CurrencyService.CanAfford(player, type, amount)
Returns true if the player has enough of the given currency.
CurrencyService.Add(player, type, amount)
Adds currency to a player's balance. Clamps to the hard cap.
CurrencyService.Spend(player, type, amount)
Deducts currency if the player can afford it. Returns true on success and false if insufficient funds.
CurrencyService.GetAllBalances(player)
Returns both balances at once as a table. Use this when sending balance updates to the client.
InventoryService
Manages item ownership, stacking, equipping, and selling. Inventory is stored as itemId to count. Equipped items are stored as slot to itemId. Returns copies of tables to prevent external mutation.
InventoryService.AddItem(player, itemId)
Adds one instance of an item to inventory. Stacks up to MAX_STACK (9999).
InventoryService.RemoveItem(player, itemId)
Removes one instance of an item. Cleans up zero-count entries automatically.
InventoryService.SellItem(player, itemId)
Removes one item and pays out its tier sell value in coins. Returns the amount earned or false on failure.
InventoryService.SellDuplicates(player)
Sells all extras keeping one of each item. Returns total coins earned.
InventoryService.EquipItem(player, itemId, slot)
Equips an item to a named slot. Validates ownership before equipping.
PityService
Tracks how many rolls a player has done per case without hitting each tier. Pity counters are stored as caseId_tierName keys. Only tiers with a pityMax set in TierConfig are tracked.
PityService.GetGuaranteedTier(player, caseId)
Returns the rarest tier the player has hit pity on, or nil if none. CaseService calls this before every roll.
PityService.Track(player, caseId, rolledTier)
Increments counters for all tiers not hit this roll and resets the counter for the tier that was hit.
PityService.GetPitySummary(player, caseId)
Returns count, max, and progress percentage per tier. Use this to render pity progress bars in the UI.
PityService.ResetAll(player)
Clears all pity counters across all cases. Call this on rebirth.
CaseService
The central orchestrator. Calls every other service in sequence to complete a full case open. Includes a per-player cooldown to prevent spam opens. If any step fails after currency is charged, the player is automatically refunded.
CaseService.OpenCase(player, caseId)
Runs the full 9-step open pipeline. Returns a result table with success, item, tier, tierData, wasGuaranteed, pitySummary, newBalances, and totalRolls.
CaseService.GetCaseData(player, caseId)
Returns all data needed to populate the case UI in one call. Call this on UI open, not on every roll.
EventService
Manages timed global events that modify game behaviour. Only one event can run at a time. Events auto-expire and CaseService reads the active modifiers on every open.
EventService.StartEvent(eventId)
Starts a named event from the event pool. Notifies all clients via EventNotify.
EventService.GetLuckMultiplier(baseLuck, caseId)
Returns the effective luck value factoring in any active luck_boost event.
EventService.GetCaseCost(baseCost, caseId)
Returns the effective case cost factoring in any active cost_discount event.
EventService.GetCurrencyMultiplier()
Returns the sell payout multiplier for any active double_currency event.
//shared modules
TierConfig
Single source of truth for all tier definitions. Readable by both server and client since it lives in ReplicatedStorage. All tier weights, colors, pity caps, and sell values are defined here. No other file should hardcode tier data, always read from TierConfig.
TierConfig.TierOrder
Ordered array of tier names from most common to rarest. Position matters — RNGService uses index 1 as the baseline when applying luck multipliers. Never shuffle this list.
TierConfig.Tiers
Table of tier definitions keyed by tier name. Each entry contains weight, color (hex string), label, pityMax (nil to disable pity for that tier), and sellValue.
TierConfig.GetRollTable()
Returns the tier table in the format RNGService.RollTier() expects. Always call this rather than building the table manually elsewhere.
TierConfig.GetTier(tierName)
Returns the full definition for a single tier by name. Returns nil with a warning if the tier does not exist.
TierConfig.GetSellValue(tierName)
Returns the coin sell value for a given tier. Used by InventoryService when a player sells an item.
TierConfig.GetPityMax(tierName)
Returns the pity cap for a given tier, or nil if that tier has no pity system.
CaseDefinitions
Pure data module. No logic, only definitions. Defines every case in the game along with the item pools for each tier inside that case. Both server services and client controllers read from here. Adding a new case requires no changes to any other file.
CaseDefinitions.Cases
Table of all case definitions keyed by case id. Each case contains name, cost, currency type, image asset id, and an items table keyed by tier name.
CaseDefinitions.GetCase(caseId)
Returns the full definition for a case by id. Returns nil with a warning if the case does not exist.
CaseDefinitions.GetItemPool(caseId, tierName)
Returns the array of items for a specific case and tier. Called by CaseService after a tier has been rolled to pick the actual item.
CaseDefinitions.GetAllCaseIds()
Returns a flat array of all case id strings. Useful for populating the case selection UI or iterating over all cases.
Types
Shared Luau type definitions for every data structure in the system. Import this module and annotate variables to enable Studio type checking and catch mismatches early. These types also serve as living documentation. If a structure changes, update it here and Studio will flag all affected code.
Types.Item
Defines the shape of a single item: id, name, image, description, and weight.
Types.TierData
Defines the shape of a tier definition: weight, color, label, pityMax (optional), and sellValue.
Types.CaseDef
Defines the shape of a full case definition: name, cost, currency, image, and items table.
Types.PityEntry
Defines the pity progress for a single tier: count, max, and progress percentage.
Types.PitySummary
A table of PityEntry values keyed by tier name. Returned by PityService.GetPitySummary().
Types.Balances
Defines the currency and gems balance table returned by CurrencyService.GetAllBalances().
Types.OpenResult
The full result table returned by CaseService.OpenCase(). Contains success, reason, item, tier, tierData, wasGuaranteed, pitySummary, newBalances, and totalRolls.
Types.PlayerData
The full data structure saved per player in DataService. Mirrors getDefaultData() exactly — if you add a field to one, add it to the other.
Types.EventData
The payload sent to clients via EventNotify when a global event starts: id, name, description, duration, and endsAt timestamp.
//adding a new case
Open CaseDefinitions and add a new entry to the Cases table following this structure. No other file needs to change.
CaseDefinitions.Cases = { -- existing cases above new_case = { name = "New Case", cost = 200, currency = "currency", -- or "gems" image = "rbxassetid://YOUR_ID", items = { Common = { { id = "new_001", name = "Item Name", weight = 100, image = "rbxassetid://0", description = "Description here" }, }, Rare = { { id = "new_002", name = "Rare Item", weight = 100, image = "rbxassetid://0", description = "Description here" }, }, -- add pools for each tier you want in this case }, }, }
▸
Item ids must be unique across all cases. Use a prefix per case to avoid collisions, for example new_001 vs starter_001.
▸
Then add a button for the new case in ClientMain inside the caseButtons table.
//adding a new event
Open EventService and add a new entry to the EVENT_POOL table. The auto-event system will pick it up automatically.
{ id = "new event", name = "New Event", description = "something happens for like 90 seconds!", duration = 90, type = "luck_boost", -- luck_boost | cost_discount | double_currency value = 2, -- multiplier or discount factor targetCase = nil, -- nil = all cases, or a specific caseId string },
▸
To trigger an event manually from the server console during testing, run require(game.ServerScriptService.Services.EventService).StartEvent("my_event")
//adding a new tier
Open TierConfig. Add the tier name to TierOrder in the correct position from most common to rarest, then add its definition to the Tiers table.
TierConfig.TierOrder = { "Common", "Uncommon", "Rare", "Epic", "Legendary", "Mythic", "Brainrot", "NewTier", -- add in rarity order } TierConfig.Tiers.NewTier = { weight = 0.01, color = "#ffffff", label = "NEW TIER", pityMax = 5000, sellValue = 100000, }
▸
TierOrder position matters. RNGService uses index 1 as most common when applying luck multipliers. Never shuffle this list.
//tips and security
▸
The client is a display layer only. It fires remotes and shows results. All logic including rolls, cooldowns, and currency changes happen server-side.
▸
Never call DataService.Set(player, "currency", x) directly from other services. Always go through CurrencyService.Add() or CurrencyService.Spend() so validation runs.
▸
Item ids must be globally unique across all cases. Use a case-specific prefix on every id.
▸
Run RNGService.Simulate(TierConfig.GetRollTable(), 10000) in the Studio server console to verify your weight distribution before shipping.
▸
If you add a new field to the default player data, add it to both getDefaultData() in DataService and the PlayerData type in Types. Existing players get the new field via the reconcile function on next load.
▸
Delete or disable DevTestUI before going live. It exposes every system directly to the player.
//troubleshooting
?
Case open returns insufficient_funds - check the player's currency balance and confirm the case cost and currency type match in CaseDefinitions
?
Pity not triggering - confirm pityMax is set on the tier in TierConfig. Common and Uncommon have nil pityMax by default
?
Data not saving - check the DataStore name in DataService. If you changed it, existing player data is under the old key. Enable API services in game settings if DataStore calls fail entirely
?
Item not appearing in inventory - verify the item id in CaseDefinitions is unique and does not collide with another case's item ids
?
Remote errors on client - confirm all remotes exist in ReplicatedStorage/Remotes with the correct type. OpenCase, GetCaseData, GetInventory, and GetBalances must be RemoteFunctions. All others must be RemoteEvents
?
Nothing happens on open - check the server output for errors. The most common cause is a service failing to require due to a wrong script path
